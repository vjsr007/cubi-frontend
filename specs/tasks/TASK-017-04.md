# TASK-017-04: Frontend EmulatorSettingsPage, EmulatorConfigPage, and UI Components

## Context
Build two frontend pages for emulator configuration:
1. **EmulatorSettingsPage** — Manages canonical "general settings" (resolution, fullscreen, language, etc.) with a tabbed sidebar per emulator, category-grouped controls, save/reset/preview workflow.
2. **EmulatorConfigPage** — Manages per-system emulator overrides (exe path, launch args, RetroArch core) stored in TOML config.

## Parent Design
- **DES**: [DES-017 — Emulator General Settings](../design/DES-017-emulator-settings.md)

## Acceptance Criteria

### EmulatorSettingsPage
- [x] Tabbed sidebar listing all 8 emulators with color-coded indicators and customized-count badges
- [x] Settings content area grouped by category (Video, Audio, System, Performance) with category headers
- [x] `SettingControl` component renders appropriate UI per setting type:
  - `bool` — toggle switch with locked indicator
  - `select` — pill button group for option selection
  - `range` — slider input with percentage display
- [x] Locked settings (V-Sync) display a lock badge and are non-interactive
- [x] Local edit tracking with "unsaved changes" indicator and dirty state
- [x] Save button batch-writes all pending edits via `set_emulator_setting` API calls
- [x] Reset button clears stored settings for the active emulator (reverts to defaults)
- [x] Preview Config button opens a modal showing the generated native config snippet
- [x] `PreviewModal` component with syntax-highlighted code block, copy-to-clipboard, emulator badge

- [x] Search/filter by system name, emulator name, or system ID
- [x] Customized count badge in top bar
- [x] Save persists overrides to TOML config via `configStore.saveConfig()`
- [x] Reset All clears all overrides

- [x] `SettingDefinition`, `EmulatorSettingValue`, `ConfigWriterInfo`, `SettingCategory` types in `src/types/index.ts`

### Key Design Decisions
- **Inline styles**: Both pages use inline `React.CSSProperties` objects for consistency with the existing codebase pattern
- **Local edit buffering**: EmulatorSettingsPage tracks edits in a `localEdits` map keyed by `"emulatorName::settingKey"`, only flushing to backend on explicit Save
- **Emulator color theming**: Each emulator has a brand-consistent color used for sidebar highlights, badges, and section headers across both pages
- **No Zustand store**: Settings state is page-local (useState) since it is not needed globally; config overrides use the existing `configStore`
