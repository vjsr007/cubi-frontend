1. Executive Summary

Cubi Frontend is a feature-rich multiplayer emulator frontend comparable to EmulationStation-DE, Pegasus, and LaunchBox. It provides a beautiful, gamepad-navigable interface for browsing, managing, and launching games across 38 gaming platforms from a single unified library.

Built on Tauri 2 (Rust backend) with React 19 (TypeScript frontend), the application delivers native performance with a modern web UI. Key highlights of v0.6.0:

  • Steam Store Integration — reviews, metadata, and game linking
  • 3 Themes — Default grid, Aurora (Xbox 360 3D carousel), HyperSpin (spinning wheel)
  • Metadata Editor — full game info editing with YouTube video search & download
  • Multi-Source Scraper — ScreenScraper, TheGamesDB, IGDB, SteamGridDB, MobyGames
  • 8 Emulator Config Writers — RetroArch, Dolphin, PCSX2, DuckStation, PPSSPP, RPCS3, xemu, Ryujinx
  • 6 Languages — English, Spanish, German, French, Japanese, Portuguese
  • Gamepad Navigation — full controller support with spatial navigation

2. Project Overview & Architecture

2.1 High-Level Architecture

React 19 UI (Pages, Components, Stores) → Tauri IPC (invoke / events) → Rust Backend (Commands → Services → DB) → SQLite

2.2 Portable Folder Architecture

All user game data lives in a single portable folder (e.g., E:\Emulation). Emulators are installed separately (e.g., via EmuDeck). The data structure:

{DATA_ROOT}/ ├── roms/{system}/ # ROM files per system (38 systems) │ ├── gamelist.xml # Metadata from scrapers │ └── downloaded_images/ # Per-game box art ├── bios/ # BIOS files ├── saves/ # Save states per emulator ├── storage/downloaded_media/ # Scraped media (13 types × 38 systems) └── tools/launchers/ # Launch scripts (.ps1 files)

2.3 Key Design Principles

  • Never assume emulators are inside the data folder
  • Support multiple data roots (USB, NAS, etc.)
  • Auto-detect EmuDeck paths, allow custom paths
  • All paths configurable via TOML config
  • Gamepad as primary input method

6. Spec-Driven Development (SDD) Methodology

6.1 Workflow Cycle

REQ (Requirement) → DES (Design) → TASK (Breakdown) → CODE → TEST → REVIEW

6.2 SDD Rules

RuleDescription
1. Never code without a specEvery feature starts as a REQ, gets a DES, then TASKs
2. Specs are source of truthIf code diverges from spec, update spec first
3. One TASK per commitEach task maps to a single, reviewable commit
4. Tests before implementationWrite test stubs in DES phase, implement in CODE phase

6.3 Spec Numbering Convention

TypeFormatExamplePurpose
RequirementREQ-XXXREQ-021What needs to be built
DesignDES-XXXDES-021How it will be implemented
TaskTASK-XXX-YYTASK-021-03Individual work units linked to DES
