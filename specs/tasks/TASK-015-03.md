# TASK-015-03: Scanner Two-Phase Refactoring

## Context
Refactor the ROM scanner to respect per-system path overrides. Systems with a custom path are scanned first (Phase 1), then the default `roms/` directory is walked for remaining systems (Phase 2).

## Acceptance Criteria
- [x] Phase 1: Load overrides from DB, scan each override path using the system's file extensions
- [x] Phase 2: Walk default `roms/` directory, skip systems already scanned in Phase 1
- [x] Extracted `scan_system_folder()` helper used by both phases
- [x] Gamelist.xml enrichment works in both phases
- [x] No duplicate scanning of the same system

## Implementation Notes

### Files Modified
- `src-tauri/src/commands/scanner.rs` — Refactored `scan_roms` command

### Algorithm
```
1. db.get_rom_path_overrides() → HashMap<system_id, path>
2. For each override:
   - Look up SystemDef by system_id
   - If custom_path exists and is_dir:
     - load gamelist.xml from custom_path
     - scan_system_folder(custom_path, system_def, db, gamelist)
     - Add system_id to scanned_systems Set
3. Walk {data_root}/roms/:
   - For each subdir, identify system
   - If system.id in scanned_systems → skip
   - Otherwise scan normally
```

### Key Design Decisions
- `scan_system_folder()` accepts a `Path` and `SystemDef`, encapsulating the WalkDir + extension filter + DB upsert logic
- `scanned_systems: HashSet<String>` prevents double-scanning
- Phase 1 silently skips invalid override paths (logged as warning) — doesn't fail the whole scan

## Test Plan
- Scan with no overrides → same behavior as before
- Set override for one system → verify it scans from custom path
- Set override to invalid path → verify it's skipped and other systems scan normally
